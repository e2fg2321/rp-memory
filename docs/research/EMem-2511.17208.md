# EMem: A Simple Yet Strong Baseline for Long-Term Conversational Memory of LLM Agents

**arXiv:** 2511.17208 (November 2025)
**Authors:** Sizhe Zhou, Jiawei Han
**Venue:** ICML

## Core Problem
Fixed context windows prevent coherent interactions across sessions. Existing approaches trade off coarse retrieval vs. fragmented views. Lossy compression discards potentially relevant info.

## Architecture: Event-Centric Memory with EDUs

### Theoretical Foundation
Neo-Davidsonian event semantics: represent meaning as "events" with multiple arguments, not independent binary relations.

### Elementary Discourse Units (EDUs)
- Self-contained statements bundling: participants, temporal info, local context
- Normalized entities, source turn attributions, minimal local context
- Example: "Bob spent five days in Tokyo in March 2024 to attend the Global AI Innovation Symposium 2024 at Tokyo University."
- Structure: e = (text(e), src(e), tau(e)) — text, source turns, timestamp

### Variant 1: EMem-G (Graph-Based)
1. **EDU Extraction** — LLM processes sessions → EDUs with metadata
2. **Event-Argument Extraction** — Second LLM pass → event type + role-argument pairs
3. **Heterogeneous Graph** — Three node types (Session, EDU, Argument), three edge types (session-EDU, EDU-argument, synonym at δ≥0.9)
4. **Dense Retrieval** — Embed query, retrieve top-K_e EDUs by cosine sim + mention detection for argument nodes
5. **Recall-Oriented LLM Filtering** — Bias toward recall, not precision
6. **Personalized PageRank** — Seed initialization from filtered nodes → PPR propagation
7. **EDU Selection & QA** — Top-K EDUs by PPR score → zero-shot CoT QA

### Variant 2: EMem (Lightweight)
- Skip graph + argument retrieval
- Just: embed query → top-K EDUs by similarity → LLM filter → QA
- Surprisingly competitive with EMem-G

### Dual Extraction for Assistant Utterances
- Atomic EDUs (fine-grained facts) + Structured Chunks (cohesive blocks with summaries)
- Summaries used for retrieval indexing; full chunks revealed only during QA

## Key Results

### LoCoMo (multi-session dialogue, ~24K tokens avg)
- EMem-G: 0.780 LLM Score (gpt-4o-mini), 0.853 (gpt-4.1-mini)
- vs FullContext: 0.723 / 0.806
- vs Nemori: 0.744 / 0.794
- Context: 738-988 tokens vs 23,653 full context

### LongMemEval_S (~105K tokens avg)
- EMem-G: 77.9% avg accuracy (gpt-4o-mini), 84.9% (gpt-4.1-mini)
- vs FullContext: 55.0% / 65.6%
- Strongest on: temporal reasoning (74.8% vs 42.1%), multi-session (73.6% vs 38.3%), knowledge-update (94.4% vs 78.2%)
- Context: 1.0K-3.6K tokens vs 101K

### Ablations
- EDU filtering is most critical component (-3.6 to -4.8% without it)
- Graph propagation has modest but targeted value (temporal, multi-hop)
- QA zero-shot CoT matters on LongMemEval (-4.5 to -5.4%)

### Weakness
- Event-centric approach under-captures user preferences/attitudes
- single-session-preference: 32.2% vs Nemori's 46.7%

## Key Takeaways for rp-memory
1. **EDUs >> relation triples** for conversational memory — preserve coherence
2. **Non-compressive representation** avoids information loss
3. **Recall-oriented filtering** > precision-oriented for memory retrieval
4. **Source turn tracking** (provenance) is baked into EDU structure
5. **30-100x context reduction** possible with good structure
6. **Event memory alone is insufficient** — needs complementary persona/preference models
7. **Graphs are optional** — simple dense retrieval + LLM filter is often sufficient
